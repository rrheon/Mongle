### 1. 전체 홍보 전략 개요

**핵심 메시지**: "매일 하나의 질문이, 멀어진 가족을 다시 연결합니다."

**채널**: 인스타그램 릴스 (30초 이내 숏폼) — AI TTS + 감성 BGM
**톤**: 따뜻한 스토리텔링. 광고가 아닌 "공감 콘텐츠"로 소비되도록 설계
**빈도**: 주 3회 릴스(월/수/금) + 보조 스토리·카루셀 2회(화/목)
**게시 시간**: 릴스 오후 7~8시 / 스토리 점심 12~1시

#### 타겟별 접근법

| 타겟 | 감정 트리거 | 접근 방식 |
|------|-----------|----------|
| 따로 사는 가족 | 그리움, 안도감 | "떨어져 있어도 매일 만나는 법" |
| 사춘기 자녀 둔 부모 | 서운함, 희망 | "말 안 하는 아이가 앱에선 3줄을 썼다" |
| 부모님께 표현 못 하는 자녀 | 미안함, 사랑 | "차마 못 한 말, 질문이 대신 꺼내줬다" |
| 일상 기록 가족 | 따뜻함, 재미 | "우리 가족만의 일기장" |
| 해외 거주 교포/유학생 | 시차 속 연결 | "바다 건너도 매일 답변이 도착한다" |

#### 필수 표기 사항 (법적/윤리적)
- 모든 가상 답변 콘텐츠: **"실제 사용 예시를 재구성한 콘텐츠입니다"** 고정 자막
- AI TTS 영상: **"AI 음성으로 제작되었습니다"** 캡션 표기
- 감정 자극 콘텐츠 후미: **"가족마다 소통 방식은 다릅니다"** 균형 문구

### 2. 릴스 시리즈 기획 (5개)

#### 시리즈 A: "사실은요"
- 컨셉: 가족에게 차마 직접 못 한 말을 고백하는 1인칭 독백체
- 타겟: 20~30대 자녀
- 확장: "사실은요, 엄마" / "사실은요, 아빠" / "사실은요, 딸에게"

#### 시리즈 B: "읽는 순간 멈춤"
- 컨셉: 몽글 질문을 보여주고 "가족에게 보낸다면?" 상상 유도
- 타겟: 범용 (바이럴 목적)
- 확장: 추억 편, 감사 편, 미래 편

#### 시리즈 C: "우리 엄마 답변"
- 컨셉: 가상의 가족 답변 재현. 화면 분할 말풍선으로 시각 처리, 음성은 내레이션만
- 타겟: 부모 / 따로 사는 가족
- 확장: "우리 아빠 답변" / "중2 아들 답변" / "시골 할머니 답변"

#### 시리즈 D: "365일 후"
- 컨셉: 1년간 몽글을 쓴 가족의 변화를 타임라인으로 보여주는 형식
- 타겟: 일상 기록 가족
- 확장: "100일 후 우리 가족" / "스트릭 50일의 기적"

#### 시리즈 E: "그날의 질문"
- 컨셉: 기념일/시즌 맞춤 질문. 시즌 마케팅 연동
- 타겟: 전 타겟
- 확장: 어버이날 편, 추석 편, 연말 편, 새학기 편

바쁜 일상 속, 가족과 주고받는 대화는 "밥 먹었어?", "별일 없지?"에서 끝나기 쉽습니다. 몽글은 매일 하나의 질문을 통해 그 이상의 이야기를 자연스럽게 꺼낼 수 있도록 도와줍니다. 어릴 적 기억, 요즘의 고민, 앞으로의 꿈, 오늘 감사한 일까지 — 평소에는 묻기 쑥스러웠던 이야기를 질문 하나로 가볍게 시작해 보세요.

### B. v2 컨셉 정의

**원칙 1 — 질문 답변이 본체, 꾸미기는 보상이다.**
배경/장비 시스템은 "매일 답변"이라는 핵심 행동을 해치면 안 된다. 꾸미기가 목적이 되어 답변이 수단으로 전락하는 설계를 피한다.

**원칙 2 — 그룹은 같은 공간을 본다.**
가족이 같은 홈 화면(같은 배경, 같은 동료 캐릭터들)을 본다는 경험이 몽글의 핵심 감정이다. 배경은 **그룹 공유**, 장비는 **개인**이 기본값.

**원칙 3 — 하트는 희소성이 있어야 한다.**
하트가 너무 잘 모이면 모든 아이템이 금방 개방되어 목표가 사라진다. 기본 획득(답변·스트릭·광고 시청)과 소비(재촉/재발급/꾸미기)의 밸런스를 먼저 시뮬레이션한 뒤 가격 확정.

#### D-4. 렌더링 규격

- **Z-order**: `배경 → 그림자 → 발밑(dec_f) → 등(dec_b) → 바디(MongleMonggle 본체) → 머리(dec_h) → 풍선/공중 트레일 → 이름라벨`.
- 슬롯별 캔버스(D-2 표 참조). 투명 PNG 또는 SwiftUI Shape.
- hop 애니메이션 동기화: 장식은 `MongleView` 외부 sibling. 본체 hop과 같은 ScenePosition binding 구독.
  - 머리: 본체와 동일 phase, 0.5pt 지연(부드러운 무게감).
  - 등: 본체와 동일 phase.
  - 발밑: **반전 위상**(본체 down일 때 펫 up) — squish 충돌 회피.
- flip(좌우 이동): 슬롯 3개 모두 본체 flip에 동기화.
- **squish 분리**: 본체 `scaleEffect(x:1.15, y:0.7, anchor:.bottom)`(Components.swift:1408)는 캐릭터 ZStack에만. 장식은 squish 미적용 + 별도 micro-bounce.
- **그림자 처리**: `dec_f_*` 장착 시 기본 ellipse 그림자 자동 hide/dim.

#### D-5. 사용자 플로우

1. 상점 → **꾸미기 탭** (라운드 1 "장비"에서 명칭 변경) → 머리/등/발밑 3-segment.

3. 카드 탭 → 풀스크린 미리보기(내 몽글 + 해당 장식 합성, 미장착 vs 장착 토글).
4. **미소유: 30초 무료 미리보기 옵션** (전환율 향상, 라운드 2 신규 D-1).
5. 미소유 구매 → 구매 직후 자동 장착 옵션 체크박스.

### H. API 설계 — **2026-04-23 장식 컨셉 반영**

```
GET  /shop/backgrounds                          # 카탈로그 + 내 그룹 소유 여부
GET  /shop/decorations?slot=head|back|foot      # 카탈로그 + 내 소유 여부 (라운드 1 /shop/equipments 폐기)
POST /shop/backgrounds/{id}/purchase            # 그룹 구매. body: { groupId }. Header: Idempotency-Key
POST /shop/backgrounds/{id}/apply               # 그룹 적용. body: { groupId }
POST /shop/decorations/{id}/purchase            # 개인 구매. Header: Idempotency-Key
PUT  /me/decoration                             # { headId?, backId?, footId? } — 서버가 owned 검증 (라운드 1 /me/equipment 폐기)
GET  /groups/{groupId}/background/active        # 홈 진입 시 현재 배경
GET  /groups/{groupId}/members/decoration       # 멤버별 장착 상태 일괄 (멤버별 개별 호출 금지, §C-5')
POST /rewards/admob/ssv                         # AdMob Server-Side Verification 콜백 (하트 +1)
GET  /me/hearts/history                         # 일일 한도 확인용 (1일 3회 광고 상한 서버측 카운터)
```

- 모든 구매 응답은 `ApiResponse { ok, data: { remainingHearts, owned: [...] } }` 래퍼 준수.
- 실패 케이스: `INSUFFICIENT_HEARTS`, `ALREADY_OWNED`, `NOT_OWNED_APPLY`, `NOT_OWNED_DECORATION`, `INVALID_SLOT`, `UNAVAILABLE_SEASONAL`, `GROUP_MEMBERSHIP_REQUIRED`.
- **구매 트랜잭션 (라운드 1 QA 블로커)**: `User.hearts` 차감 + `*Owned` INSERT 단일 트랜잭션. 클라이언트 `Idempotency-Key` 헤더(UUIDv4) 필수, 24시간 동일 응답.
- **AdMob SSV 필수 (라운드 1 QA 블로커)**: 클라 `onRewarded` 단독 지급 금지. AdMob → `/rewards/admob/ssv` HMAC 서명 검증 후 `User.hearts` 증가. `AdRewardLog(userId, day, count)` 테이블로 일 3회 상한 관리(서버 UTC).
- 장착 API는 서버가 `UserDecorationOwned` 포함 여부 검증, 미소유 장식 장착 차단.
- 그룹 적용/구매 요청은 요청 유저의 `FamilyMembership` 재확인(탈퇴 후 구 토큰 악용 방지).
- 장식 변경 동기화는 §C-5' 정책: **푸시 미발사**, 다음 포그라운드 진입 시 `GET /groups/{groupId}/members/decoration` 일괄 fetch.
