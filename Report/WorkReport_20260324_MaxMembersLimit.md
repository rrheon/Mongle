## 2. 근거 — MongleScene 수용 가능 인원 계산

### 2.1 MongleSceneView 배치 파라미터

`Components.swift`의 `MongleSceneView`에 정의된 물리 상수:

| 파라미터 | 값 | 의미 |
|----------|----|------|
| `collisionRadius` | 76pt | 캐릭터 간 최소 중심-중심 거리 |
| `wallPadding` | 50pt | 씬 경계 여백 |
| `overlapLimit` | 10 | 겹침 허용 한계 (배치 시도 횟수) |

배치 알고리즘은 **최대 30회 랜덤 시도 후 강제 배치**하므로 인원이 많아질수록 캐릭터가 겹침.

---

### 2.2 씬 유효 면적 계산

**기준: iPhone 15 Pro (393 × 852pt)**

```
전체 화면 높이        852pt
TopBar (헤더+질문카드) -176pt  (헤더 56+60 safeArea + 질문카드 60)
TabBar               - 83pt
────────────────────────────
씬 높이              593pt
```

```
유효 배치 영역 (캐릭터 중심 기준):
  가로: 393 - wallPadding×2 = 393 - 100 = 293pt
  세로: 593 - wallPadding×2 = 593 - 100 = 493pt
```

---

### 2.3 캐릭터 수용 계산

#### 방법 A — 격자 배치 (최적 배치 상한)

```
열 수 = floor(293 / 76) = 3열  (여백 포함 실질 4열 가능)
행 수 = floor(493 / 76) = 6행

격자 기준 최대 = 4 × 6 = 24명  (이론적 상한, 현실적으로 불가)
```

#### 방법 B — 원넓이 기반 (랜덤 배치 현실적 추정)

```
유효 면적     = 293 × 493 = 144,449 sq pt
캐릭터 점유   = π × 76²  =  18,146 sq pt
랜덤 배치 효율 ≈ 80%  (격자 대비 랜덤 배치 손실)

수용 인원 = (144,449 / 18,146) × 0.8 ≈ 6.4명
```

#### 방법 C — 실용 여유 고려

랜덤 배치 알고리즘의 30회 시도 한계와 씬 양끝 여백을 감안하면, **비겹침 상태로 안정적으로 배치 가능한 현실 수치는 6~7명** 수준.

---

### 2.4 최종 수치 결정: 8명

| 관점 | 수치 | 비고 |
|------|------|------|
| 수학적 수용 한계 | ~6.4명 | 방법 B |
| 이론적 최적 배치 | ~8명 | 방법 A 하한 |
| 일반 가족 규모 | 4~8명 | 통계적 가족 구성 |
| Preview 기본값 | 5명 | 디자이너 의도 |

- 6명은 다소 타이트하고 `int`로 표현 시 의미가 약함
- 7명은 비대칭적
- **8명**: 이론 최적 배치 하한과 일치하고, 직계 가족(부모+자녀 최대 6명)+조부모 2명 커버 가능. 2의 거듭제곱으로 명확한 인상 제공.

## 3. 구현 내용

**파일**: `FamTreeServer/src/services/FamilyService.ts`

```typescript
const MAX_GROUPS = 3;
const MAX_MEMBERS = 8; // MongleScene 수용 한계 기반 (collisionRadius=76pt, 유효 씬 면적 기준)
```

`joinFamily()` 내에 그룹 입장 전 인원 체크 추가:

```typescript
// 그룹 최대 인원 확인
const currentMemberCount = await prisma.familyMembership.count({
  where: { familyId: family.id },
});
if (currentMemberCount >= MAX_MEMBERS) {
  throw Errors.conflict(`그룹 인원이 가득 찼습니다. 최대 ${MAX_MEMBERS}명까지 참여할 수 있습니다.`);
}
```

- 체크 위치: 이미 멤버인지 확인 직후, 트랜잭션 시작 이전
- 에러 코드: `409 Conflict`
- 에러 메시지: 최대 인원 수 포함 (클라이언트 표시용)

## 4. 추가 고려 사항

### 향후 UI 개선 제안

현재는 서버에서만 제한하고 있어 사용자가 초대 코드 입력 → 가입 시도 → 에러를 겪는 흐름임.
이후 iOS/Android에서 아래를 추가하면 UX 개선 가능:

- 그룹 상세 화면에 `"현재 N/8명"` 인원 표시
- 8명 가득 찬 그룹의 초대 버튼 비활성화 또는 "가득 참" 배지 표시

### 상수 관리

`MAX_GROUPS`, `MAX_MEMBERS` 모두 `FamilyService.ts` 최상단에 함께 선언되어 있어, 수치 변경 시 한 곳에서 관리 가능.
