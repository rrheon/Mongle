## 1. 데이터 기반 개선 사이클 — Analytics 도입 가이드

### 1-1. 도구 선택 — Firebase Analytics 권장

| 항목 | Firebase Analytics | Amplitude |
|---|---|---|
| 비용 | 무료 (BigQuery 연동도 GCP 무료 티어 가능) | 월 10M 이벤트 무료, 그 이상 유료 |
| 통합 난이도 | 낮음 — Mongle 이미 FCM/Crashlytics 사용 중이라 SDK 추가만으로 통합 | Middle |
| Funnel/Cohort | DebugView + 펑션 페이지 + BigQuery로 가능 | UI 친화도 높음 |
| **결정** | **Firebase Analytics 권장** — 이미 firebase-admin/Crashlytics 사용 중인 인프라와 일치 | |

### 1-2. 핵심 측정 이벤트 — 몽글 핵심 funnel "진심 답변 작성"

```
[열람] question_view  → [작성 진입] answer_compose_open  → [완료] answer_submit_success
                                          ↓
                              [이탈 추적] answer_compose_abandon (with reason)
```

#### 심어야 할 커스텀 이벤트 (이름은 snake_case 30자 이하 — Firebase 제약)

| 이벤트 | 파라미터 | 설명 |
|---|---|---|
| `daily_question_view` | `question_id`, `member_count` | 매일 나오는 질문 노출 |
| `answer_compose_open` | `question_id`, `entry_point` (push/home/notif) | 작성 화면 진입 |
| `answer_compose_keystroke` | `question_id`, `length_bucket` | 30/60/120자 임계 — 어디서 멈추는지 |
| `answer_compose_abandon` | `question_id`, `last_length`, `reason` (back/kill/bg) | **핵심 — 이탈 지점** |
| `answer_submit_success` | `question_id`, `length`, `time_ms_to_submit` | 완료 시점 + 작성 소요 시간 |
| `nudge_send` | `target_user`, `nudge_type` | Peer nudge 사용 빈도 |
| `notification_open` | `notification_type`, `delivery_to_open_ms` | 푸시 → 앱 진입 시간 |
| `family_invite_share` | `share_method` (link/copy/messenger) | 그룹 공유 채널 |

### 1-3. 면접 스토리 템플릿 — "데이터로 본 의사결정"

> 가설 후 측정 → 임시저장 도입 → 결과 확인의 사이클로 검증.

**문제 정의**
> "몽글의 핵심 가치는 가족 간 진심 답변 누적입니다. 그러나 작성 화면 진입 후 완료율이 낮을 거라는 가설이 있었고, 이를 검증하기 위해 Firebase Analytics를 도입했습니다."

**측정·분석**
> "BigQuery export로 연결해 `answer_compose_open` 대비 `answer_submit_success` 비율을 측정했더니 X%였습니다. `answer_compose_abandon`에 last_length 파라미터를 함께 심어둔 덕분에 평균 X자 시점에서 이탈한다는 사실을 알 수 있었습니다."

**개선 액션**
> "(a) **자동 임시저장** — 30초 간격으로 SwiftData/Room에 draft를 저장하고, 다음 진입 시 복원. (b) **글자 수 가이드 라벨** 제거 — 부담을 주는 카운터를 숨기고 placeholder만 노출. (c) **앵커링** — 첫 줄에 가족 멤버 이름을 자동 prefill하여 시작 부담을 낮춤."

**결과**
> "출시 2주 차 기준 완료율 X% → Y%로 개선. 동시에 평균 작성 글자수도 늘어 답변 품질도 향상됐습니다."

### 1-4. 구현 체크리스트 (3개 코드베이스 공통)

- [ ] **iOS**: `FirebaseAnalytics` SPM 추가 → `Analytics.logEvent(_:parameters:)` 래핑한 `MongleAnalytics` 단일 진입점.
- [ ] **Android**: `firebase-bom` BOM에서 `firebase-analytics-ktx` 추가 → `Firebase.analytics.logEvent(...)` 래퍼.
- [ ] **공통 이벤트 스키마 문서화**: `docs/analytics-events.md` (서버/iOS/Android 동일 이름 강제).
- [ ] **DebugView 활성화** 후 모든 이벤트가 똑같은 이름·파라미터 형식으로 들어오는지 QA.
- [ ] **BigQuery export** 활성화 — 무제한 funnel/cohort 분석.
- [ ] **PII 점검** — `user_id`는 hashed UUID로만, 답변 내용·이메일·실명 절대 금지.

## 2. 크래시 대응 / 안정화 트러블슈팅 케이스

> Jira에는 직접적인 "Crashlytics fatal" 이슈는 거의 없으나, **잠재적 크래시·런타임 경고·무한로딩·thread-safety 위반**을 미리 잡은 안정화 작업이 풍부합니다. 포트폴리오에서는 이를 "사후 대응"이 아닌 "선제적 안정화" 스토리로 풉니다.

### 사례 ① MG-83 — 배지/푸시/백그라운드 안정화 (Android)

**증상 (요약)**
- OS 앱 아이콘 배지가 인앱 미읽음 알림 수와 분리되어 동기화되지 않음 (배지 기능 자체 미구현).
- **신규 푸시 type 도착 시 silent fail** — `when` 분기에 `else` fallback 없어 잠재적 크래시 위험.
- 비로그인 상태에서 푸시 tap 후 다른 사용자로 로그인하면 stale `pendingNotificationType`이 자동 소비.
- `NotificationScreen` 진입 시 `unreadCount`가 매 composable 재실행마다 O(n) 재계산.

**원인 (iOS 패리티 분석)**
- 1년 전 iOS에서 동일 클래스 이슈(MG-39/55/61)를 해결한 경험을 Android에 이식.
- Compose recomposition 라이프사이클과 `setBadgeCount(0)` 무조건 호출의 간섭.

**해결**
- `NotificationCompat.setNumber()` 도입 + mark-read/delete/markAllAsRead 직후 `syncAppIconBadge()` 호출.
- 푸시 type `when` 분기 7가지 + `else` 안전 fallback.
- `unreadCount`를 stored property로 변경, mutation 시점만 갱신.
- `pendingNotificationType=null` 명시적 cleanup.

**검증 시나리오**
- 새 푸시 도착 시 OS 아이콘 배지에 미읽음 수 표시.
- 알림 1건 mark-read 시 배지 즉시 -1, deleteAll 시 0.
- **테스트로 unknown type 푸시 발송 시 크래시 없이 안전 진입**.

**임팩트**
- 잠재적 ANR/크래시 카테고리 6건을 단일 PR로 일괄 차단.

---

### 사례 ② MG-14 — "그룹 선택·알림 진입 시 무한로딩"

**증상**
- 그룹 선택 화면에서 개별 그룹 진입 또는 알림에서 진입 시 로딩 인디케이터가 무한히 돌고 화면 전환 안 됨.

**원인 분석 과정**
1. **재현 조건 좁히기** — 콜드스타트 직후, 푸시 tap 진입에서만 재현.
2. **로깅** — TCA store action sequence 추적 → `refreshHomeData`가 두 번 dispatch 되어 두 번째 effect가 첫 번째를 cancel.
3. **확정** — APNs cold start에서 store nil 상태로 액션 enqueue → 인증 완료 후 재진입에서 재dispatch → race.

**해결**
- store cold start sentinel 도입 + pending action queue (MG-47 일환).
- `refreshHomeData` effect에 cancellation token 부여, 중복 시 fresh 재시작.

**임팩트**
- 사용자 첫인상 결정 화면에서의 멈춤 현상 제거.

---

### 사례 ③ MG-101 — `SimpleDateFormat` thread-safety 위반 (Android)

**증상 (잠재적 크래시)**
- `NotificationScreen`에서 알림 목록을 빠르게 스크롤할 때 간헐적으로 `NumberFormatException` 또는 잘못된 날짜가 표시됨.

**원인**
- `SimpleDateFormat`은 thread-safe하지 않음. Compose recomposition이 다른 dispatcher에서 동일 인스턴스를 동시에 호출해 내부 `Calendar` 상태가 깨짐.

**해결**
- `java.time.format.DateTimeFormatter`로 전면 교체 (immutable, thread-safe).
- ApiQuestionRepository.parseDate에 UTC TimeZone 명시(MG-100) — 1일 어긋남 동시 수정.

---

### 사례 ④ MG-90 — 토큰·광고 race 안정화 (Android, iOS MG-34/35/37 패리티)

**문제 영역**
- `RefreshCoordinator` 동시 호출로 refresh token이 두 번 사용되어 server에서 invalidate.
- Keychain 이중 저장소(legacy + new) 사이의 정합성 깨짐.
- logout 실패 시 토큰 잔존으로 다음 사용자가 이전 사용자 세션으로 진입.

**해결**
- `Mutex` 기반 single-flight refresh + 진행 중 호출자에게 동일 결과 반환.
- 단일 저장소로 통합, 마이그레이션 스크립트.
- logout best-effort + 실패 시 사용자에게 명시적 재로그인 유도.

**임팩트**
- 인증 관련 잠재적 크래시·세션 오염 시나리오 일괄 차단.

---

### 사례 ⑤ MG-94 — OkHttpClient timeout 누락 → ANR 방지

**증상**
- 약전계 환경에서 네트워크 요청이 무한 hang → 사용자가 앱 강제종료.

**원인**
- OkHttpClient 기본값은 timeout 무한.

**해결**
- `connectTimeout(10s) / readTimeout(15s) / writeTimeout(15s)` 명시.
- TokenAuthenticator의 raw string interpolation을 Moshi 직렬화로 교체(MG-93) — JSON injection 방지 동시 수정.

---

### 핵심 메시지 (면접용)

> "Crashlytics fatal로 직접 떠오르는 것을 잡는 게 아니라, **iOS에서 먼저 해결한 클래스의 이슈를 Android에 패리티로 이식**하면서 동시성·thread-safety·timeout 같은 잠재적 크래시 카테고리를 한 번에 차단했습니다. 한 PR이 6~8건의 Jira 티켓을 동시에 closing 한 사례가 다수 있고, 이는 단순 fix가 아니라 **양 플랫폼을 같은 정신모델로 운영**한 결과입니다."
