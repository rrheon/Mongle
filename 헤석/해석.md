## 1. UserDefaultsManager

```swift
private let userDefaults: UserDefaults
private let encoder: JSONEncoder
private let decoder: JSONDecoder
```

**질문**: 이런 걸 코드에서 각각 생성해서 쓰는 것과 위와 같이 다 선언해놓고 사용하는 것의 차이?

### 답변

차이 4가지:

**① 성능 (사실은 큰 차이 X)**

`JSONEncoder`/`JSONDecoder`는 인스턴스 생성 비용이 작아 매번 만들어도 체감 차이 거의 없음. **여기선 성능이 핵심 이유가 아님.** "성능 때문에 미리 만들어둔다"는 흔한 오해.

**② 의존성 주입 (DI) 가능성 — 진짜 이유**

미리 선언해두면 init에서 외부가 주입 가능한 구조가 만들어짐:

```swift
init(
    userDefaults: UserDefaults = .standard,
    encoder: JSONEncoder = JSONEncoder(),
    decoder: JSONDecoder = JSONDecoder()
) {
    self.userDefaults = userDefaults
    // ...
}
```

지금은 init이 encoder/decoder를 외부에서 안 받지만, **언제든 받을 수 있는 구조**. 예:
- 테스트에서 특정 `dateDecodingStrategy`를 가진 decoder를 주입하고 싶을 때
- 어떤 환경에선 `.prettyPrinted` 출력을 원할 때

메서드 안에서 `JSONEncoder()`를 매번 만들면 **밖에서 손쓸 방법이 없음**. 프로퍼티로 선언해두면 init 한 줄만 바꿔서 주입 가능.

**③ 설정 일관성**

만약 인코더에 특정 설정을 줘야 한다면 (예: `outputFormatting = .sortedKeys`), 매번 만들면 모든 메서드에서 같은 설정을 반복해야 함. 한 번 만들어두면 한 군데서 끝.

APIClient.swift의 decoder가 정확히 이 케이스 — ISO8601 dateDecodingStrategy를 init에서 한 번 셋업:
```swift
self.decoder = JSONDecoder()
decoder.dateDecodingStrategy = .custom { ... }
```
매 request 메서드에서 반복하면 일관성 깨지고 버그 위험.

**④ 테스트 가능성**

프로퍼티로 보유하면 테스트에서 가짜 객체 주입이 쉬움. `UserDefaults`도 `init(userDefaults: UserDefaults = .standard)` 덕분에 테스트에선 `UserDefaults(suiteName: "test")` 같은 격리된 인스턴스를 넣을 수 있음. 이게 안 되면 테스트가 글로벌 `UserDefaults.standard`를 오염시켜 서로 간섭.

### 격언

> "프로퍼티로 미리 선언하는 진짜 이유는 **성능이 아니라 의존성 주입 + 설정 일관성 + 테스트 가능성**. 매번 생성은 외부가 손쓸 수 없는 닫힌 코드."

## 2. APIClient

### (A) `APIClientProtocol`을 어떻게 사용하는가

`AnswerRepository.swift`에 정답이 있음:

```swift
final class AnswerRepository: AnswerRepositoryInterface {
    private let apiClient: APIClientProtocol               // ← protocol 타입으로 받음

    init(apiClient: APIClientProtocol = APIClient.shared) {
        self.apiClient = apiClient
    }

    func create(_ answer: Answer, moodId: String?) async throws -> Answer {
        let endpoint = AnswerEndpoint.create(...)
        let dto: AnswerDTO = try await apiClient.request(endpoint)   // ← protocol 메서드만 호출
        return AnswerMapper.toDomain(dto)
    }
}
```

**이게 Phase 2.3에서 본 generic vs existential의 실전 사례**. 여기선 `private let apiClient: APIClientProtocol`로 **existential**을 골랐음. 왜?

- Repository 인스턴스 하나에 **APIClient든 MockAPIClient든** 들어가야 함 → 정적 타입이 둘 다 받아야 함
- Repository를 generic으로 `<C: APIClientProtocol>`로 만들 수도 있지만, Repository 사용처에서 타입 파라미터가 줄줄이 따라다녀 가독성 떨어짐
- `apiClient.request(...)` 호출은 핫패스가 아니라 ms 단위 네트워크 호출 — witness dispatch 비용이 0에 가깝게 묻힘
- ⇒ 이런 상황이 **"existential을 의도적으로 고르는" 정상 케이스**. APIClientProtocol에 `Sendable` 만 붙어있고 Self/AT 요구사항이 없어 existential로 받기에도 깔끔.

테스트에선 `AnswerRepository(apiClient: MockAPIClient())` 하면 끝. `MockAPIClient`도 같은 protocol을 채택하므로 Repository 코드는 한 줄도 안 바뀌고 가짜 응답을 줄 수 있음.

### (B) 401 처리 공통화 (라인 98–103 / 129–135)

본인 통찰이 정확함. **거의 똑같은 패턴이 두 번 반복됨**:

```swift
// 라인 98-108 (request<T>)
if response.statusCode == 401 {
    let newToken = try await self.attemptTokenRefresh()
    var retryRequest = try endpoint.buildURLRequest()
    retryRequest.setValue("Bearer \(newToken)", forHTTPHeaderField: "Authorization")
    let (retryData, retryResponse) = try await self.perform(retryRequest)
    try self.validateResponse(retryResponse, data: retryData)
    do {
        return try self.decoder.decode(T.self, from: retryData)   // ← 여기만 다름
    } catch {
        throw APIError.decodingError(error.localizedDescription)
    }
}

// 라인 129-136 (request without response)
if response.statusCode == 401 {
    let newToken = try await self.attemptTokenRefresh()
    var retryRequest = try endpoint.buildURLRequest()
    retryRequest.setValue("Bearer \(newToken)", forHTTPHeaderField: "Authorization")
    let (retryData, retryResponse) = try await self.perform(retryRequest)
    try self.validateResponse(retryResponse, data: retryData)
    return                                                          // ← 여기만 다름
}
```

**두 블록의 95%가 동일**. 차이는 "디코딩해서 반환하느냐 / 그냥 반환하느냐"뿐. 공통 추출 충분히 가능.

**추천 리팩토링**:

```swift
/// 401 응답 시 토큰 갱신 + 재시도. 재시도 응답의 (data, response)를 반환.
private func handle401AndRetry(_ endpoint: APIEndpoint) async throws -> (Data, HTTPURLResponse) {
    let newToken = try await self.attemptTokenRefresh()
    var retryRequest = try endpoint.buildURLRequest()
    retryRequest.setValue("Bearer \(newToken)", forHTTPHeaderField: "Authorization")
    let (data, response) = try await self.perform(retryRequest)
    try self.validateResponse(response, data: data)
    return (data, response)
}
```

호출부:

```swift
// request<T>
if response.statusCode == 401 {
    let (retryData, _) = try await handle401AndRetry(endpoint)
    do { return try decoder.decode(T.self, from: retryData) }
    catch { throw APIError.decodingError(error.localizedDescription) }
}

// request (no response)
if response.statusCode == 401 {
    _ = try await handle401AndRetry(endpoint)
    return
}
```

**이득**:
- 중복 제거. 401 로직 변경 시 한 곳만 고침
- 향후 "401 처리에 로깅 추가" 같은 요구사항이 와도 한 군데
- 두 메서드의 핵심(디코딩 여부)만 남아 가독성 ↑

**주의**: token refresh는 MG-42 코멘트(APIClient.swift 46-49줄)에 따르면 과거 사고가 있던 영역. 리팩토링은 별도 PR로 분리 + 충분한 수동/단위 테스트 후 머지 권장.

추가 관찰: `endpoint.buildURLRequest()`도 양쪽에서 두 번씩 호출됨. `request` 진입 시 한 번 만든 걸 재사용하는 게 자연스러울 텐데 매번 빌드하는 건 의도가 있을 수도 (URLRequest를 inout으로 변형하느라). 리팩토링 시 같이 검토할 가치.

## 4. NetworkMonitor의 `@unchecked Sendable` — 왜? actor 안 되나?

본인 추측이 거의 맞음. 정확히 풀어 설명:

### `@unchecked Sendable`이 뭔지부터

- **`Sendable`**: "이 타입의 인스턴스는 여러 스레드/Task 사이를 안전하게 건너다닐 수 있다"는 컴파일러 약속
- 일반적으로 컴파일러가 "프로퍼티들이 다 immutable이거나 Sendable인지" 자동 검증해서 OK/NG 판정
- **`@unchecked Sendable`**: "검증을 못 하니까 **내가 책임지고** Sendable이라고 선언한다" — 컴파일러에게 도장 찍어달라고 강제 요청

NetworkMonitor는 다음 이유로 컴파일러가 자동 검증 불가:
1. `_isConnected: Bool` 가 var (mutable) → 자동 검증 fail
2. `NWPathMonitor` 자체가 Sendable이 아닐 수 있음
3. NSLock + DispatchQueue로 직접 동기화 관리 중

그래서 **개발자가 "NSLock으로 안전하게 처리했으니 내 책임으로 Sendable이라 선언"** 한 게 `@unchecked Sendable`의 의미. "Trust me bro" 모드.

### 왜 actor로 안 바꿨나

actor로 바꿀 수 **있음**. 하지만 **그 순간 외부 API가 전부 깨짐**.

```swift
// 현재 (final class + @unchecked Sendable):
networkMonitor.isConnected            // 동기 — 즉시 true/false 반환

// 만약 actor로 바꾸면:
await networkMonitor.isConnected       // 비동기 — await 필요!
```

actor의 모든 프로퍼티/메서드 접근은 자동으로 `async`가 됨 (actor isolation). 그래서:

```swift
// APIClient.checkConnectivity() — 현재 동기 + throws
private func checkConnectivity() throws {
    guard networkMonitor.isConnected else {
        throw APIError.offline
    }
}
```

이게 actor 전환 시:
```swift
private func checkConnectivity() async throws {   // ← async 전염!
    guard await networkMonitor.isConnected else {
        throw APIError.offline
    }
}
```

→ `request` 진입 직후 첫 줄에 `try checkConnectivity()`가 있는데 이게 `try await`로 바뀌면서, **호출자(Repository → Feature)까지 줄줄이 async 전염**. 단순한 "연결 상태 즉시 확인"이 actor hop으로 지연됨.

### 결정의 트레이드오프

| | `@unchecked Sendable` (현재) | `actor` |
|---|---|---|
| 외부 인터페이스 | 동기 (`isConnected` 즉시) | 비동기 (`await` 필요) |
| 컴파일러 검증 | ❌ (개발자 책임) | ✅ |
| 동기화 코드 | NSLock 수동 | actor 자동 |
| 성능 | 락 한 번 (수 ns) | actor hop (수 μs) |
| 적용 범위 | 이 파일 안에서 끝 | 호출자 전체가 async로 |

→ "**연결 상태는 즉시 알아야 하고, 호출자 입장에서 sync로 보이게 하고 싶다**"는 의도가 `@unchecked Sendable` 선택의 진짜 이유.

### 안전성 검토

코드를 보면:
- `_isConnected`, `_connectionType` 두 var는 **반드시 lock 안에서만 읽고 씀**
- 읽기 path (`isConnected`, `connectionType` getter): lock.lock() → defer unlock
- 쓰기 path (`pathUpdateHandler` 클로저): lock.lock() → ... → lock.unlock()
- 모든 mutable state 접근이 `NSLock`으로 보호됨 → `@unchecked Sendable` 약속 이행 OK

**위험 신호**: 만약 누군가 새로운 mutable 프로퍼티를 추가하면서 lock으로 보호하는 걸 깜빡하면 race condition 발생하는데, **컴파일러가 잡아주지 않음**. `@unchecked`의 본질적 위험. 그래서 이런 클래스엔 **"이 클래스에 var 추가할 땐 반드시 NSLock으로 감쌀 것"** 같은 주석을 큼지막하게 박아두는 게 좋음. 지금은 그게 없는 상태.

### 격언

> "`@unchecked Sendable` = '나는 actor의 자동 보호를 안 받겠지만 수동으로 lock 걸어 안전을 보장한다'는 개발자 선언. **동기 인터페이스를 유지하려는 명확한 이유**가 있을 때만 쓸 것. 그 외엔 actor가 안전."

## 5. MongleData에서 Repositories는 apiClient에 endpoint를 던져주는 역할만 하는가?

**거의 맞지만 약간 더 있음.** 정확히 짚으면:

### Repository가 실제로 하는 일 (AnswerRepository 예)

```swift
func create(_ answer: Answer, moodId: String?) async throws -> Answer {
    let endpoint = AnswerEndpoint.create(...)              // ① endpoint 생성
    let dto: AnswerDTO = try await apiClient.request(endpoint)   // ② API 호출
    return AnswerMapper.toDomain(dto)                       // ③ DTO → Entity 변환
}
```

세 가지 책임:
1. **Endpoint 만들기** — Domain의 Entity/파라미터를 API 호출 형태로 가공
2. **API 호출** — apiClient에 위임
3. **DTO → Domain 변환** — Mapper 호출

→ 즉 **"endpoint 던지는 역할만"이 아니라 "endpoint 만들기 + 호출 위임 + 결과 변환"** 의 3단 어댑터.

### 그리고 가끔 비즈니스 조합도

`AnswerRepository.swift` 라인 36-47:

```swift
func getByDailyQuestion(dailyQuestionId: UUID) async throws -> [Answer] {
    let endpoint = AnswerEndpoint.getFamilyAnswers(...)
    let response: FamilyAnswersResponseDTO = try await apiClient.request(endpoint)
    var answers = response.answers.map { AnswerMapper.toDomain($0) }
    if let myAnswer = response.myAnswer {
        let myDomain = AnswerMapper.toDomain(myAnswer)
        if !answers.contains(where: { $0.id == myDomain.id }) {
            answers.insert(myDomain, at: 0)            // ← 내 답변을 맨 앞으로
        }
    }
    return answers
}
```

→ 서버 응답이 `myAnswer`와 `answers`를 분리해서 주는데, Domain 입장에선 **"내 답변까지 포함한 단일 배열"** 이 자연스러우니 Repository에서 합쳐줌. **"서버 모양 ↔ Domain 모양 변환"** 의 한 형태고, 단순히 endpoint 던지는 게 아닌 **약간의 조합 로직**.

또 라인 56-60 `hasUserAnswered`도 비슷:
```swift
func hasUserAnswered(...) async throws -> Bool {
    let answer = try await getByUserAndDailyQuestion(...)
    return answer != nil    // 같은 클래스 다른 메서드를 재사용해 derived 정보 계산
}
```

### 정확한 정의

> "Repository는 **서버(또는 다른 데이터 소스)와 Domain 사이의 어댑터**. 주된 일은 'endpoint 호출 + Mapper 변환'이지만, 서버 모양과 Domain 모양이 어긋날 때 **얇은 조합/필터링 로직**도 여기서 처리한다."

### 해선 안 되는 것

- UI 로직 (그건 Feature/Reducer로)
- 비즈니스 규칙의 복잡한 분기 (그건 Use Case나 Domain Service로 — Mongle엔 아직 없음)
- 캐싱 같은 큰 책임이 추가되면 Repository 안이 거대해질 수 있으니, 그땐 LocalDataSource + RemoteDataSource로 분리하는 게 정석

### Mongle의 Repository 패턴 평가

지금은 매우 **얇은 Repository (thin repository)**. CRUD와 약간의 조합만. **나쁘다는 게 아니라 학습용 출발점으로 가장 깨끗한 형태**. 앱이 커지면서 캐싱·오프라인 동기화·복합 비즈니스 규칙이 필요해질 때 자연스럽게 살이 붙는 구조라 확장성도 OK.

## 6. Repository에서 API 호출만 하고 실제 API 동작은 어디서?

**질문**: Repository는 `apiClient.request(endpoint)`로 위임하던데, 실제 네트워크 동작은 어디서 일어나는가?

### 호출 스택 — 위에서 아래로

```
[Repository]                  apiClient.request(endpoint)
    │ "endpoint 좀 호출해줘"
    ▼
[APIClient.request()]         ① 연결 확인
                              ② URL 만들기 (endpoint.buildURLRequest)
                              ③ 토큰 첨부 (attachAuthToken)
                              ④ ★ perform(request) ← 여기서 진짜 호출
                              ⑤ 401이면 토큰 갱신 후 재시도
                              ⑥ 응답 검증 (validateResponse)
                              ⑦ JSON 디코딩
    │
    ▼
[APIClient.perform()]         try await session.data(for: request)
    │                          ↑ 이 한 줄이 "진짜 네트워크 호출"
    ▼
[URLSession]                  Foundation 표준 HTTP 클라이언트
    │                          (Apple이 만든 것, Mongle 코드 아님)
    ▼
[OS 네트워크 스택]             · TCP 소켓 열기
                              · TLS 핸드셰이크 (HTTPS)
                              · HTTP 요청 전송
                              · 응답 패킷 수신
    │
    ▼
서버 (AWS API Gateway → Lambda → DB)
```

→ **실제 동작이 일어나는 한 줄은 `APIClient.perform()` 안의 `session.data(for: request)`**.

### 코드로 핀포인트

```swift
// APIClient.swift 152-162줄
private func perform(_ request: URLRequest) async throws -> (Data, HTTPURLResponse) {
    do {
        let (data, response) = try await session.data(for: request)
        //                          ↑
        //                          이 한 줄이 "진짜 네트워크 호출".
        //                          여기서 OS에 HTTP 요청이 나가고 응답을 받아옴.
        guard let httpResponse = response as? HTTPURLResponse else {
            throw APIError.invalidResponse
        }
        return (data, httpResponse)
    } catch let urlError as URLError {
        throw mapURLError(urlError)
    }
}
```

이 한 줄이 끝나기 전까지 `await`이 멈춰 있고, 응답이 도착하면 다음 줄이 실행됨.

### URLSession이 뭐냐

**어원**: `URL` + `Session`. "URL 단위의 통신 세션"이라는 뜻. **Apple Foundation 프레임워크가 제공하는 HTTP 클라이언트** (cURL이나 axios의 Swift 버전이라고 보면 됨).

**역할**: 우리가 작성하는 코드와 OS의 네트워크 스택 사이의 다리. 우리는 "이 URLRequest를 보내줘"라고만 말하고, URLSession이:
- TCP 연결 풀 관리
- TLS 핸드셰이크 (HTTPS면 자동)
- HTTP 헤더 직렬화
- 응답 파싱
- 쿠키/캐시 처리
- 타임아웃 관리

를 다 알아서 해줌. Mongle은 이걸 한 번 더 감싸서 토큰 첨부·재시도·디코딩 같은 **앱 공통 정책**만 추가한 게 APIClient.

**`session.data(for:)`**: URLSession의 메서드. Swift 5.5+에서 추가된 async/await 버전. 이 메서드 하나가:
1. URLSession에게 요청 위임
2. URLSession이 OS에 요청
3. OS가 서버에 요청
4. 응답 도착할 때까지 await
5. 응답을 Data + HTTPURLResponse로 묶어 반환

까지 다 처리.

### 각 계층의 책임 정리

| 계층 | 위치 | 책임 |
|---|---|---|
| **Repository** | `MongleData/Repositories/...` | endpoint 만들기 + 호출 위임 + 결과 변환 |
| **APIClient** | `MongleData/.../APIClient.swift` | 토큰·재시도·401·디코딩 같은 **앱 공통 정책** |
| **URLSession** | Apple Foundation (외부) | HTTP 프로토콜 처리 + 연결 관리 |
| **OS 네트워크 스택** | iOS/macOS 커널 | TCP/TLS/패킷 송수신 |
| **서버** | AWS | 비즈니스 로직 + DB |

→ **"실제 API 동작"의 의미를 어디까지 잡느냐**에 따라 답이 달라짐:
- HTTP 요청 자체를 만들고 보내는 것 = **`APIClient.perform()`의 한 줄**
- 그걸 더 깊이 보면 = **URLSession이 그 뒤에서 다 함**
- 서버에서 일어나는 것 = **AWS Lambda/DB 영역** (iOS 코드 밖)

### 한 줄 격언

> "Repository는 **endpoint를 만들 뿐**, 실행은 APIClient에게 위임한다. APIClient는 **앱 공통 정책(토큰·재시도·401)을 둘러쌀 뿐**, 진짜 HTTP 요청은 URLSession에게 위임한다. URLSession은 **HTTP 프로토콜만 알 뿐**, 실제 패킷 송수신은 OS에게 위임한다. → 각 층은 **자기 책임만** 지고 그 아래로 위임하는 구조."

### 같은 패턴이 곳곳에 — 디스패치의 일반화

이 "**위임 사슬**" 구조가 좋은 소프트웨어 설계의 핵심. Phase 2.3에서 본 PWT를 통한 dispatch도 같은 본질 — 호출자는 추상만 보고, 구체는 한 단계 아래에서 처리. 사슬의 어느 한 마디만 바꿔 끼우면 동작이 바뀌지만(테스트의 MockAPIClient, 다른 백엔드 전환 등), 다른 마디는 모르고 살 수 있음.

### 7-A1. KeychainTokenStorage (AuthRepository.swift 148–239줄)

#### 핵심 코드
```swift
protocol TokenStorageProtocol {
    func saveToken(_ token: String) throws
    func saveRefreshToken(_ token: String) throws
    func getToken() -> String?
    func getRefreshToken() -> String?
    func clearToken()
    func clearRefreshToken()
}

final class KeychainTokenStorage: TokenStorageProtocol {
    private let service = "com.mongle.auth"
    private let tokenKey = "auth_token"
    private let refreshTokenKey = "refresh_token"

    private func save(_ value: String, forKey key: String) throws {
        let data = Data(value.utf8)
        let query: [CFString: Any] = [
            kSecClass: kSecClassGenericPassword,
            kSecAttrService: service,
            kSecAttrAccount: key,
            kSecAttrAccessible: kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly
        ]
        let updateAttributes: [CFString: Any] = [kSecValueData: data]
        let updateStatus = SecItemUpdate(query as CFDictionary, updateAttributes as CFDictionary)

        if updateStatus == errSecItemNotFound {
            var addQuery = query
            addQuery[kSecValueData] = data
            let addStatus = SecItemAdd(addQuery as CFDictionary, nil)
            guard addStatus == errSecSuccess else {
                throw KeychainError.saveFailed(addStatus)
            }
        } else if updateStatus != errSecSuccess {
            throw KeychainError.saveFailed(updateStatus)
        }
    }
}
```

#### 왜 중요한가
UserDefaults는 **평문 저장**. 인증 토큰을 거기 두면 jailbreak된 단말에선 그대로 노출됨. **민감 데이터(토큰/비밀번호/PIN)는 Keychain에 저장하는 게 iOS의 정석**.

#### 알아야 할 두 가지

**① C API 래퍼 — Swift답지 않지만 표준**
- `SecItemAdd`, `SecItemUpdate`, `SecItemDelete`, `SecItemCopyMatching` 4개 함수가 핵심
- 인자는 `CFDictionary` — Swift `Dictionary`를 `as CFDictionary`로 캐스트
- Key는 `kSecClass`, `kSecAttrService`, `kSecAttrAccount` 같은 CFString 상수
- 오래된 C 스타일이라 거칠지만, 이게 Apple 보안 API의 표준 표현

**② `kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly` — 보안 등급 옵션**

Keychain 항목의 접근 정책 결정. 옵션 비교:

| 옵션 | 의미 |
|---|---|
| `kSecAttrAccessibleWhenUnlocked` | 기기 잠금 해제 상태에서만 접근. 백그라운드 푸시 처리 시 막힘 |
| `kSecAttrAccessibleAfterFirstUnlock` | 부팅 후 첫 잠금 해제 한 번 거치면 접근 가능. 백그라운드 OK |
| `kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly` ← Mongle 선택 | 위와 같음 + **iCloud Keychain 동기화 X + 백업으로 다른 기기 이전 X** |
| `kSecAttrAccessibleAlways` | 항상 접근 (잠금 무관) — 보안 약함, deprecated |

**왜 `ThisDeviceOnly`인가**: 인증 토큰은 **이 기기에서만 유효**해야 함. iCloud로 다른 사용자 기기에 동기화되면 토큰 노출 위험. `ThisDeviceOnly`가 인증 토큰의 표준 선택.

#### Update-or-Add 패턴
```swift
let updateStatus = SecItemUpdate(...)
if updateStatus == errSecItemNotFound {
    SecItemAdd(...)
}
```
Keychain엔 "upsert"가 없음. 무조건 Update 시도 → 없으면 Add. 이게 일반 패턴.

#### 학습 포인트
> "민감 데이터 = Keychain, 평범한 설정 = UserDefaults. Keychain의 거친 C API는 외워두는 편이 빠름. `kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly`가 인증 토큰의 표준 접근 정책."

### 7-A2. TokenRefreshCoordinator — actor 실전 (APIClient.swift 28–41줄) ⭐

#### 핵심 코드
```swift
private actor TokenRefreshCoordinator {
    private var inFlight: Task<String, Error>?

    func refresh(using block: @Sendable @escaping () async throws -> String) async throws -> String {
        if let existing = inFlight {
            return try await existing.value  // 다른 호출자의 결과 공유
        }
        let task = Task<String, Error> { try await block() }
        inFlight = task
        defer { inFlight = nil }
        return try await task.value
    }
}
```

#### 왜 중요한가 (Swift 6 동시성 학습 관심사와 정확히 일치)
**actor 이론은 알아도 "언제 어떤 모양으로 쓰는지"가 어려운데, 이게 교과서적 예시.**

#### 발생 가능한 문제 (코드 17–27줄 주석에 실제 사고 기록)
콜드 스타트 + 푸시 도착 + scenePhase active 가 거의 동시에 발생하면:
1. 여러 API 호출이 동시에 401을 받음
2. 각자 `attemptTokenRefresh()` 호출
3. 서버에 refresh 요청이 동시 다발로 도착
4. 서버가 "동일 refresh token으로 여러 요청 = replay 공격"으로 감지
5. → 강제 로그아웃 (MG-42 사고)

#### actor가 해결하는 방식
**actor 안의 메서드는 한 번에 하나씩만 실행됨** (actor isolation). 그래서 `refresh()` 안의 코드가 atomic하게 처리:
- 첫 호출자: `inFlight == nil` 확인 → Task 시작 → `inFlight`에 저장 → `await task.value`
- 두 번째 호출자: actor가 첫 호출이 끝날 때까지 메서드 진입 대기 → 진입 시점엔 `inFlight`에 Task 있음 → 같은 `existing.value`를 await
- → **N 명의 동시 호출자가 한 번의 refresh 결과를 공유**. 서버엔 refresh 요청 1번만 감.

#### 미묘한 부분 — `Task<String, Error>?` 핸들 패턴
이전 구현은 `isRefreshing: Bool + refreshResult: Result?` 두 필드였음. 그런데 race가 있었음:
- 첫 대기자가 결과 소비 + `refreshResult = nil`로 초기화
- 그 직후 두 번째 대기자가 깨어나서 `refreshResult`를 봤더니 nil → `.unauthorized` throw

→ `Task<String, Error>?`를 in-flight 핸들로 보관하면 **Task의 결과가 Task 내부에 캐싱**됨. 여러 await가 같은 결과를 race 없이 공유. `defer { inFlight = nil }`로 Task 종료 후 핸들을 비워 다음 401 주기에서 새 refresh 가능.

#### 학습 포인트 (NetworkMonitor와 대조!)

| | NetworkMonitor | TokenRefreshCoordinator |
|---|---|---|
| 패턴 | `final class + @unchecked Sendable + NSLock` | `actor` |
| 외부 인터페이스 | 동기 (`isConnected`) | 비동기 (`refresh() async`) |
| 선택 이유 | 동기 인터페이스 유지 + 즉시 응답 | 직렬화 보장 + 자연스러운 비동기 흐름 |

> "actor를 쓸 자리: **비동기 흐름이 자연스럽고 + 직렬화를 보장해야 하는** 상태/연산. 단순 변수 보호는 NSLock + @unchecked로도 가능하지만, 'in-flight 요청 공유' 같은 복잡한 직렬화는 actor가 정답."

### 7-A3. APIError — 에러 모델 설계 사례

#### 핵심 코드
```swift
public enum APIError: Error, Equatable {
    case offline
    case timeout
    case invalidURL
    case invalidResponse
    case networkError(String)
    case decodingError(String)
    case serverError(statusCode: Int, message: String?)
    case unauthorized
    case notFound
    case unknown

    public var isRetryable: Bool {
        switch self {
        case .timeout, .networkError: return true
        case .serverError(let code, _): return code >= 500
        default: return false
        }
    }

    public var requiresLogin: Bool {
        self == .unauthorized
    }
}

extension APIError: LocalizedError {
    public var errorDescription: String? { localizedDescription }
}
```

#### 왜 중요한가
**앱 전반의 에러 처리 흐름의 척추**. 이걸 모르면 다른 코드의 `catch APIError.unauthorized` 같은 분기가 안 읽힘.

#### 알아야 할 3가지

**① 비즈니스 의미 단위로 case 분리**
단순한 "Error 받았다"가 아니라 **"어떤 종류의 실패인지"** 가 타입으로 표현됨. 호출자가 정확히 어떤 case인지 패턴 매치로 분기 가능.

**② 재시도 가능 여부를 case 메서드로 (`isRetryable`)**
```swift
public var isRetryable: Bool {
    switch self {
    case .timeout, .networkError: return true
    case .serverError(let code, _): return code >= 500  // 5xx만 재시도
    default: return false
    }
}
```
APIClient의 `withRetry`가 이 플래그를 보고 자동 재시도 여부 결정:
```swift
catch let apiError as APIError where apiError.isRetryable && attempt < maxRetryCount {
    // 백오프 후 재시도
}
```
→ **재시도 정책을 에러 타입 자체에 묶어둠**. 비즈니스 룰이 한 곳에 모임.

**③ `LocalizedError` conformance — 사용자 메시지 통합**
```swift
extension APIError: LocalizedError {
    public var errorDescription: String? { localizedDescription }
}
```
주석에 적힌 함정이 중요:
> `localizedDescription` 프로퍼티만 정의해도 Error가 NSError로 브리지될 때 Foundation이 자동 메시지 ("작업을 완료할 수 없습니다. (...오류 N)") 를 만들어 사용자에게 노출하는 문제를 막는다. `LocalizedError.errorDescription`을 구현해야 시스템 다이얼로그/`error.localizedDescription` 모두에서 우리 한국어 메시지가 나온다.

→ Swift의 흔한 함정: `error.localizedDescription`이 갑자기 영문 시스템 메시지로 바뀌는 케이스. 해결은 `LocalizedError` 채택 + `errorDescription` 구현.

#### 학습 포인트
> "에러 모델은 (1) **비즈니스 의미 단위로 case 분리**, (2) **정책(재시도/로그인 필요)을 case 메서드**로, (3) **`LocalizedError`로 사용자 메시지 통합**. 이 세 가지 패턴을 갖추면 다음 앱에서 그대로 베껴 써도 됨."

### 7-B1. AuthRepository의 `defer` 패턴

#### 핵심 코드
```swift
func logout() async throws {
    defer {
        tokenStorage.clearToken()
        tokenStorage.clearRefreshToken()
    }
    let endpoint = AuthEndpoint.logout
    try await apiClient.request(endpoint)
}

func deleteAccount() async throws {
    defer {
        tokenStorage.clearToken()
        tokenStorage.clearRefreshToken()
    }
    let endpoint = AuthEndpoint.deleteAccount
    try await apiClient.request(endpoint)
}
```

#### 왜 중요한가
**서버 호출 성공/실패와 무관하게 로컬 토큰은 반드시 정리**. 주석:
> "네트워크 오류로 logout API가 실패해도 사용자 의도는 '로그아웃'이며, 토큰을 남겨두면 다음 실행에서 만료 토큰으로 401 → sessionExpired 무한 루프가 발생할 수 있다."

#### `defer`의 진짜 용도
Swift `defer`는 함수 종료 시점에 항상 실행됨 — 정상 return, throw 발생, return 분기 어느 쪽이든.

흔한 오해: "이건 그냥 try-finally 같은 거잖아". 그런데 진짜 가치는:
- **함수 어디서 throw가 나든 정리 보장**
- **다음 줄에서 throw 가능성이 있는 코드 위에 미리 적어두면 가독성 ↑** — "이 함수 끝날 땐 무조건 토큰 정리"라는 의도가 함수 맨 위에 선언됨
- C++의 RAII, Go의 defer, Python의 `finally`와 같은 역할

#### 학습 포인트
> "**성공/실패 어느 쪽이든 반드시 해야 하는 정리 작업**은 `defer`로. 함수 맨 위에 적어두면 '이 함수는 종료 시 무조건 X를 함'이라는 보증이 시각적으로 명확해짐."

### 7-B2. MongleData.swift — 팩토리 함수와 type erasure 실전

#### 핵심 코드
```swift
public func makeAuthRepository() -> any AuthRepositoryInterface {
    AuthRepository()
}
public func makeAnswerRepository() -> any AnswerRepositoryInterface {
    AnswerRepository()
}
// ... 9개 비슷한 모양
```

#### 왜 중요한가 — 두 가지 패턴

**① 구체 타입은 internal로 숨기고 protocol만 public 노출**

`AuthRepository` 클래스 선언:
```swift
final class AuthRepository: AuthRepositoryInterface {  // ← 접근 한정자 없음 = internal
```
→ 외부 모듈(MongleFeatures, Mongle 앱 타겟)에서 `AuthRepository()`로 직접 인스턴스 못 만듬. `makeAuthRepository()`라는 팩토리 함수가 **유일한 진입점**이고, 반환 타입은 `any AuthRepositoryInterface` (existential).

⇒ **caller는 protocol만 보고, 구현체는 모름**. Phase 2.3에서 본 existential의 정상적 사용 사례 — "여기서 구현체 타입을 의도적으로 숨긴다"는 API 경계.

**② 의존성 주입의 출발점**

TCA의 `@Dependency`가 이 팩토리들을 내부적으로 호출해서 Reducer에 끼워줌. Stage 2(Feature 코드)에서 보게 될:
```swift
@Dependency(\.answerRepository) var answerRepository
```
이 `\.answerRepository` 키가 가리키는 게 결국 `makeAnswerRepository()`. `AppDependencies.swift`에 등록되어 있음.

#### type erasure가 여기서 빛나는 이유
Phase 2.3 응용에서 AnyDrawable wrapper를 만들었던 거 기억나죠? 여기선 **이미 `any` 키워드로 Swift가 자동 type erasure** 해줌. 별도 wrapper 안 만들어도 됨.

만약 `AuthRepositoryInterface`에 associated type이 있었다면 `any`로 못 받음 → 직접 `AnyAuthRepository` wrapper를 만들어야 했을 것. **Mongle Domain의 protocol들은 의도적으로 Self/AT 요구사항을 안 둠** — 그래서 existential로 깔끔하게 받을 수 있음. 이게 설계 선택의 결과.

#### 학습 포인트
> "팩토리 함수 = (1) **구체 타입 캡슐화** + (2) **DI 진입점**. `any P` 반환은 caller에게 'protocol만 보세요'라는 API 설계 의도. Domain protocol에 Self/AT를 안 두는 게 이 패턴을 가능하게 함."

### 7-B3. ISO8601Parser — silent bug 함정 + 방어 패턴

#### 핵심 코드 (14줄짜리 작은 파일)
```swift
func parseISO8601(_ string: String) -> Date {
    let formatter = ISO8601DateFormatter()
    // 밀리초 포함 형식 먼저 시도
    formatter.formatOptions = [.withInternetDateTime, .withFractionalSeconds]
    if let date = formatter.date(from: string) { return date }
    // 밀리초 미포함 형식 폴백
    formatter.formatOptions = [.withInternetDateTime]
    return formatter.date(from: string) ?? Date()
}
```

#### 왜 중요한가 — 한 가지 silent bug의 교과서적 사례

**함정**:
- 서버(Prisma)가 `"2026-04-13T12:00:00.000Z"` (밀리초 포함)을 주는데
- `ISO8601DateFormatter` 기본 옵션은 밀리초를 **거부**
- → 파싱 실패 → 코드에서 `?? Date()`로 폴백 → **"오늘 날짜"로 잘못 표시되는데 에러는 안 남**
- 사용자: "왜 12월 글이 오늘로 보이지?" / 개발자: 로그에 아무것도 안 남음 → 디버깅 지옥

#### 방어 패턴 두 가지

**① 옵션 두 단계 시도**
밀리초 포함 형식을 먼저 시도하고, 실패하면 미포함으로 폴백. 서버가 어느 쪽 형식을 보내든 대응.

**② `Date()` 폴백은 했지만 주의 필요**
`?? Date()`로 폴백한 부분은 **silent failure 그 자체**. 사용자가 잘못된 날짜를 보게 됨. 더 안전한 방향은:
- `Date?`를 반환해서 호출 측이 처리하도록 강제
- 또는 throw해서 명시적 실패로 전환

지금 코드는 "어쨌든 화면이 깨지진 않게" 우선순위로 `Date()`를 폴백. 트레이드오프. 주석으로 의도를 적어두는 게 그래서 중요.

#### 학습 포인트
> "**API 통합에서 날짜 파싱은 항상 두 번 봐라**. 서버가 주는 형식과 클라이언트의 기본 옵션이 어긋나면 silent bug. 폴백 값(`?? Date()` 같은)이 합리적인지 매번 자문. 더 안전한 방향은 Optional 반환이나 throw."

## 8. AuthRepository — socialLogin을 generic으로 빼면?

**질문**: `socialLogin(with credential: any SocialLoginCredential)`을 아래처럼 generic으로 바꾸는 게 비용 측면에서 낫지 않나? 컴파일 에러는 안 남.

```swift
func socialLogin<T: SocialLoginCredential>(with credential: T) async throws -> SocialLoginResult
```

### 비용 측면은 본인 통찰이 정확

Phase 2.3에서 본 generic vs existential의 비용 차이:
- existential → witness dispatch + (큰 값이면) boxing
- generic → specialization으로 비용 0

→ **이론적으로 generic이 더 빠름** ✅. 본인 통찰 정확.

### 그런데 여기서 그 차이가 의미 있는가? — 아니요

**5가지 이유로 `any`가 합리적**:

#### ① 호출 빈도 — 핫패스가 아님
- `socialLogin`은 사용자가 로그인할 때 **세션당 1번** 호출
- 그 안에서 일어나는 일: 네트워크 요청 1번 (수백 ms)
- existential의 witness dispatch 비용은 nanosecond 단위
- → **측정 불가능한 수준의 이득**. specialization 효과가 네트워크 호출의 그림자에 묻힘

#### ② Caller 사용성 — `any`가 더 자연스러움

caller(LoginFeature 등)는 보통 이렇게 씀:
```swift
var credential: any SocialLoginCredential
switch provider {
case .apple:  credential = AppleLoginCredential(...)
case .google: credential = GoogleLoginCredential(...)
case .kakao:  credential = KakaoLoginCredential(...)
}
try await authRepository.socialLogin(with: credential)
```

caller가 **provider별 분기 끝에 `any SocialLoginCredential`로 들고 있을 가능성이 압도적**. generic 매개변수에 `any` 값을 넘기는 건 Swift 5.7+에선 implicit opening으로 가능하지만, **caller의 시그니처가 더 복잡해짐**.

#### ③ Mock 구현 사용성 — `any`가 더 쉬움

테스트용 `MockAuthRepository`를 만들 때:
```swift
final class MockAuthRepository: AuthRepositoryInterface {
    var capturedCredential: (any SocialLoginCredential)?

    func socialLogin(with credential: any SocialLoginCredential) async throws -> SocialLoginResult {
        capturedCredential = credential   // 받은 인자를 저장해 검증
        return ...
    }
}
```

만약 protocol이 `<T: SocialLoginCredential>`이라면:
- Mock의 `socialLogin<T>(with: T)` 안에서 T를 저장하려면 → **결국 `any`로 erasure** 필요
- 즉 **generic은 호출 흐름엔 좋지만 저장이 필요한 곳엔 결국 `any`가 필요**
- → 일관성을 위해 처음부터 `any`가 단순

#### ④ Protocol requirement에 generic을 넣을 때의 제약

만약 `AuthRepositoryInterface`가:
```swift
public protocol AuthRepositoryInterface: Sendable {
    func socialLogin<T: SocialLoginCredential>(with credential: T) async throws -> SocialLoginResult
}
```

이러면:
- `any AuthRepositoryInterface`로 인스턴스를 들고 다닐 때 generic 메서드 호출이 Swift 5.7+ implicit opening에 의존
- Swift 6에서 protocol 요구사항의 generic 메서드는 추가 제약이 늘어남 (existential 호환성)
- **`MongleData.swift`의 팩토리 `makeAuthRepository() -> any AuthRepositoryInterface`** 가 깨질 수도 있음

#### ⑤ 의미론적 — 둘이 표현하는 의도가 미묘하게 다름
- `any SocialLoginCredential` — "어떤 종류의 credential이든 한 번에 한 종류 받는다. 내부에선 type erased로 다룸"
- `<T: SocialLoginCredential>` — "한 호출에 단일한 T 타입만. 그 T는 caller가 결정"

socialLogin 내부는 결국 `credential.fields`와 `credential.providerType`만 꺼내서 API에 넘김:
```swift
let endpoint = AuthEndpoint.socialLogin(
    provider: credential.providerType.rawValue,
    fields: credential.fields
)
```
→ 내부에서 T의 구체 타입이 필요 없음. protocol 두 프로퍼티만 읽으면 끝. **이런 경우 generic의 이득은 0**.

### 격언

> "**generic은 (1) 핫패스이거나 (2) caller에서 자연스럽게 구체 타입을 들고 있을 때** 이득. 한 번 호출되는 로그인처럼 비용이 네트워크에 묻히고, caller가 이미 existential로 들고 있는 경우엔 **`any`가 일관성·사용성·Mock 친화성 모두에서 우위**."

### 결론
- 컴파일 에러 안 남 ✅
- 비용 이론 본인 말이 맞음 ✅
- 하지만 실전 이득은 0이고 사용성은 손해
- **현재 코드(`any`) 유지가 정답**

## 9. DailyQuestionRepository — getCompleted/getIncomplete 합치기

**질문**: 74~82줄 두 메서드를 하나로 만드는 게 낫지 않나?

```swift
func getCompletedByFamily(familyId: UUID) async throws -> [DailyQuestion] {
    let all = try await getHistoryByFamily(familyId: familyId, limit: nil)
    return all.filter { $0.isCompleted }
}

func getIncompleteByFamily(familyId: UUID) async throws -> [DailyQuestion] {
    let all = try await getHistoryByFamily(familyId: familyId, limit: nil)
    return all.filter { !$0.isCompleted }
}
```

### 답: 합치는 것보다 더 큰 문제가 보임

먼저 합치는 방법부터, 그 다음에 진짜 더 큰 문제까지 짚을게요.

### ① 합치는 3가지 선택지

**(a) enum 매개변수 — 가장 깔끔**
```swift
enum CompletionFilter { case completed, incomplete, all }

func getByFamily(familyId: UUID, filter: CompletionFilter = .all) async throws -> [DailyQuestion] {
    let all = try await getHistoryByFamily(familyId: familyId, limit: nil)
    switch filter {
    case .completed:  return all.filter { $0.isCompleted }
    case .incomplete: return all.filter { !$0.isCompleted }
    case .all:        return all
    }
}
```
- 장점: 호출 측 의도가 명확. 새 필터(.recent 등) 추가도 자연스러움
- 단점: 호출이 약간 길어짐

**(b) Bool 매개변수 — boolean trap 주의**
```swift
func getByFamily(familyId: UUID, completedOnly: Bool) async throws -> [DailyQuestion]
// 호출 측: getByFamily(familyId: id, completedOnly: true)
```
- 단점: `true`/`false`만 봐선 의도가 불분명. "completedOnly: false가 incomplete만? 아니면 전체?" 같은 의문.
- **추천 X**

**(c) Extension default impl — 인터페이스는 깨끗하게** ⭐ 추천
```swift
public protocol DailyQuestionRepositoryInterface {
    func getHistoryByFamily(familyId: UUID, limit: Int?) async throws -> [DailyQuestion]
    // ... raw method만 protocol 요구사항으로
}

public extension DailyQuestionRepositoryInterface {
    func getCompletedByFamily(familyId: UUID) async throws -> [DailyQuestion] {
        try await getHistoryByFamily(familyId: familyId, limit: nil).filter { $0.isCompleted }
    }
    func getIncompleteByFamily(familyId: UUID) async throws -> [DailyQuestion] {
        try await getHistoryByFamily(familyId: familyId, limit: nil).filter { !$0.isCompleted }
    }
}
```
- 장점: protocol 요구사항은 `getHistoryByFamily` 하나만. 편의 메서드는 default로 제공 → **모든 구현체가 자동으로 같은 동작**. 구현체 중복 코드 0. Mock 만들 때도 raw method만 mock하면 끝.
- 함정: protocol extension 메서드는 PWT에 안 올라가 override 안 됨 (Phase 2.3 진단1). 여기선 override할 일 없는 편의 메서드라 안전.

→ **내 추천**: **(c) Extension default impl**. 깨끗하고 중복 없으며 protocol 요구사항도 단순.

### ② 진짜 더 큰 문제 — 매번 모든 데이터를 가져와서 클라이언트에서 필터링

본인 질문의 두 메서드는 둘 다:
```swift
let all = try await getHistoryByFamily(familyId: familyId, limit: nil)
return all.filter { ... }
```

`limit: nil` → **모든 히스토리를 다 다운로드 후 클라이언트에서 거름**. 만약 가족이 1년 동안 365개 질문을 받았다면 그걸 다 받아서 일부만 사용. 게다가 `getCompletedByFamily`와 `getIncompleteByFamily`를 같은 화면에서 둘 다 호출하면 **같은 데이터를 두 번 다운로드**.

#### 개선 방향 두 가지

**(가) 서버 측 필터링**
서버 API에 쿼리 파라미터 추가:
- `GET /daily-questions/family/{id}?completed=true`
- 서버가 DB에서 필터링 후 일부만 응답 → 네트워크 사용량 ↓ + 응답 속도 ↑
- 서버 작업 필요

**(나) 클라이언트 캐싱**
같은 함수가 짧은 시간 안에 두 번 호출되면 두 번 다운로드 — 캐싱 레이어 추가:
```swift
private var historyCache: (familyId: UUID, fetchedAt: Date, data: [DailyQuestion])?

func getHistoryByFamily(familyId: UUID, limit: Int?) async throws -> [DailyQuestion] {
    if let cached = historyCache,
       cached.familyId == familyId,
       Date().timeIntervalSince(cached.fetchedAt) < 60 {  // 60초 TTL
        return cached.data
    }
    // 네트워크 호출 + 캐시 갱신
}
```
- TTL은 의미에 따라 조정
- 무효화(다른 답변 작성 시 등) 처리 필요 → Repository가 거대해질 신호 → LocalDataSource + RemoteDataSource 분리 단계로 진화

### 우선순위 추천

1. **즉시 가치 있는 변경**: (c) Extension default impl로 중복 제거. 리스크 0
2. **중기 개선**: 서버 측 필터링 추가 (서버 작업 필요)
3. **장기 개선**: Repository 캐싱 레이어 → DataSource 분리

### 격언
> "**중복 코드는 표면 신호**. 합치는 방법은 여러 가지지만, 합치기 자체보다 **'왜 같은 데이터를 두 번 다운로드하는가'** 같은 더 깊은 문제가 보이는지 자문할 것. 진짜 비용은 메서드 개수가 아니라 네트워크 호출 횟수."

        t4:  요청 A: 결과 소비("토큰X") + refreshResult = nil 로 초기화  ⚠
        t5:  요청 B: 폴링 깨어남 → refreshResult 봤더니 nil → .unauthorized throw ← race!

      문제의 본질: "두 필드(isRefreshing, refreshResult)를 한 묶음으로 다뤄야 atomic한데,
      사이사이 다른 호출자가 끼어들 수 있는 자리가 너무 많음." 첫 대기자가 결과 소비 +
      초기화하는 그 짧은 순간에 두 번째 대기자가 nil을 보면 끝.

      [Task<String, Error>? 패턴의 핵심]
      · Task = "결과를 한 번 계산하고 영원히 들고 있는 상자"
      · task.value 를 await 하면 → 계산 중이면 기다림 / 끝났으면 즉시 캐시된 값 반환
      · 따라서 첫 대기자가 결과를 받았다고 Task가 사라지지 않음
      · Task 자체가 결과 저장소 → 별도의 "result nil 초기화" 단계 불필요
      · → race가 발생할 자리 자체가 사라짐

      [격언]
      "두 필드를 보호하는 락보다, 두 필드를 합쳐서 하나의 atomic한 객체로 만드는 게 더 안전."
      Task가 그 '하나의 객체' 역할.

  2. if let existing = inFlight { return try await existing.value } — 두 번째 호출자가 첫 호출자의 결과를 어떻게 공유하는가? Task.value는 여러 번 await해도 되는지?
  - Task 결과는 내부캐시된다고 했으니 이를 공유하는것같음. Task.value는 여러번 await해도 되는지에 대한 여부는 concurrency에 취소전파와 연관이 있을거 같은데 잘 모르겠음

    - 보강: ⭕ 결론 맞음 / "취소 전파" 가설은 잘못
      결론(여러 awaiter가 안전하게 await 가능) 정답. 다만 "취소 전파와 연관"은 다른 영역.

      [Task.value의 정체]
      · Task의 비동기 프로퍼티: var value: Success { get async throws }
      · 호출 의미: "이 Task가 끝날 때까지 기다린 후 결과 반환"
      · 끝났으면 즉시 캐시된 결과 반환. 아직이면 awaiter로 등록 → 결과 도착 시 깨어남
      · awaiter는 여러 명일 수 있음. Swift 런타임이 결과 도착 시 모든 awaiter를 깨움
      · → "여러 번 await 가능"이 Task의 본질적 보장 (캐시 + multi-awaiter 동시 지원)

      [취소 전파는 다른 축]
      · Task.cancel() = Task에게 "중단하라" 협조 요청. 실제 중단은 Task 코드가
        Task.isCancelled 체크하거나 cancellation point(예: Task.sleep)를 만났을 때 일어남
      · 부모-자식 관계가 있으면 부모 취소 시 자식들도 자동 취소 → 이게 "취소 전파"
      · 이건 await 횟수와 무관. 1번 await든 100번 await든 취소 동작은 동일

      [비유로 정리]
      Task.value = "택배 도착 알림 받기 (대기자 명단에 이름 적기)". 명단에 이름 여러 개여도
      택배는 한 번만 옴.
      취소 전파 = "택배 자체를 취소하면 모든 대기자가 통보 받음".
      → 두 메커니즘이 분리됨. 같은 Task에 작용하지만 다른 축.

      한 줄: "여러 await 가능성은 Task의 기본 보장. 취소와는 별개 축."

  3. defer { inFlight = nil } — 왜 함수 끝에서 핸들을 비우나? 비우지 않으면 어떤 일이 생기나?
  - inFlight 변수에 Task가 계속 담겨있는 상태 기존 Task를 계속 가지고 있는채로 다른곳에서 실행가능성이 있음
  defer를 통해 함수가 모두 종료된 후 비워줘야함

    - 보강: △ 절반 맞음 / "다른 곳에서 실행 가능성"을 정확히 풀어야
      'in-flight'(=비행 중) 핸들의 의미를 정확히 짚어야 완성.

      [defer를 빼면 일어나는 일 — 무한 401 루프]
      defer { inFlight = nil } 을 빼고 함수가 끝나면 inFlight에는 이미 완료된(또는 실패한)
      Task가 그대로 남아있음. 그 상태에서 다음 401이 발생하면:

        func refresh(...) async throws -> String {
            if let existing = inFlight {
                return try await existing.value   // ⚠ 이미 끝난 Task의 결과를 또 반환!
            }
            // 새 Task 만드는 분기에 도달하지 못함
        }

      → 이미 만료됐을 수도 있는 첫 번째 refresh 결과(토큰)를 두 번째 401에서도 반환
      → 그 토큰이 서버에서 회수됐다면 또 401 → 또 같은 캐시된 토큰 반환
      → 무한 401 루프

      [정확한 표현]
      · "in-flight" = "비행 중". Task가 진행 중일 때만 의미 있는 핸들
      · Task 종료 = 비행 끝 = 핸들 비워야 다음 비행을 받을 준비됨
      · 비우지 않으면 다음 호출자가 "비행 중인 Task가 아직 있구나" 착각 → 끝난 Task를
        await → 즉시 캐시된 (오래된) 결과 반환 → 새 refresh가 영원히 발생하지 않음

      [격언]
      "in-flight 핸들은 진행 중인 동안만 유효해야 한다. 종료 시 반드시 비우지 않으면,
      다음 호출자가 죽은 결과를 신선한 결과로 착각한다."

  4. attemptTokenRefresh(212~251줄)는 Coordinator를 어떻게 쓰는가? — refreshCoordinator.refresh { ... } 안에 들어간 클로저가 진짜 refresh 로직. 그 클로저는 어떻게
  직렬화되는가?
  - 잘 모르겠음,  코드를 살펴봣는데 refresh 함수의 결과를 기다리고 반환함 (함수 밖에서 실행될 수 있고 다른 스레드나 actor로 넘어갈 수 있는 함수), 리프레시 토큰을 얻는 함수를 실행 -> 토큰 재발급 엔드포인트 획득 -> 얻은 엔드포인트로 URL요청함수실행 (URLRequest 반환) -> 얻은 URLRequest 실행 -> 이런식인데 어떻게 직렬화 되는가? -> refresh 함수에서 클로져 block을 try await로 기다리는 task에 넣음 그리고 inFlight에 넣음으로 직렬화되고 있음

    - 보강: ❌ 핵심을 놓침 — 가장 중요한 학습 포인트
      클로저 안의 흐름 추적은 OK. 하지만 "직렬화"의 메커니즘은 다른 곳에 있음.

      [직렬화의 진짜 출처 — actor isolation]
      TokenRefreshCoordinator가 actor라는 게 핵심. actor의 본질적 보장:
        "actor의 메서드는 한 번에 하나만 실행됨.
         두 번째 호출자가 들어오면 첫 번째가 끝날 때(또는 await로 suspend될 때)까지 자동 대기."

      이 자동 대기 메커니즘이 직렬화의 근본. 코드 한 줄도 안 적었는데 컴파일러가 보장.

      [시간순으로 풀어보면]
      시간  호출자 A                     호출자 B                    inFlight
      ─────────────────────────────────────────────────────────────────────
      t1   refresh{block} 호출            -                           nil
      t2   actor 진입 OK                  refresh{block} 호출          nil
      t3   inFlight==nil 확인             actor 대기 (A가 사용중)       nil
      t4   Task 생성, inFlight=task       여전히 대기                   task ←
      t5   "await task.value" 진입         여전히 대기                  task
            → A는 suspend 상태
      t6   [A suspend됨]                  actor 진입 OK ✅             task
      t7   -                              inFlight==nil? → false        task
            (실제 클로저 실행은 Task 내부에서 별도로 진행 중)
      t8   -                              existing.value await           task
                                            → B도 suspend
      ── 네트워크 응답 도착 시점 ──────────────────────────────────────────
      t9   Task 완료, "토큰X" 결과 캐시     Task 완료, "토큰X" 결과 캐시   task
      t10  A 깨어남, "토큰X" 받음          B 깨어남, "토큰X" 받음          task
      t11  defer 실행, inFlight=nil       defer 실행 (idempotent)        nil
      t12  refresh 종료                    refresh 종료

      [핵심 두 가지]
      ① actor isolation이 진입을 직렬화
         t2에서 B가 호출했을 때 A가 actor 안에 있으니 자동 대기. 이게 첫 번째 직렬화.
         코드 명령이 아니라 actor의 본질.

      ② inFlight 검사가 클로저 실행을 중복 방지
         t6에서 B가 진입할 땐 이미 inFlight에 Task 있음 → 클로저는 A의 Task에서
         단 한 번만 실행됨. B는 그 Task의 결과를 공유받을 뿐. 이게 두 번째 직렬화.

      ["직렬화"의 정확한 정의 (여기서)]
      "N명의 동시 호출자가 있어도, 클로저(=실제 refresh 네트워크 요청)는 단 한 번만 실행되고,
       N명 모두가 그 한 번의 결과를 받는 상태."

      이걸 달성하려면:
        (a) 진입 자체가 순서대로 처리돼야 함 → actor가 해결
        (b) 두 번째 진입자가 "이미 누가 일하고 있는지" 알아야 함 → inFlight: Task<...>? 가 해결
        (c) 두 번째 진입자가 같은 결과를 받아야 함 → Task.value의 캐시 성질이 해결

      세 부분이 합쳐져야 직렬화 완성. inFlight = task 한 줄이 직렬화의 전부가 아니라,
      actor + Task 캐시 + inFlight 검사 세트의 합작.

      [비유로 굳히기 — 은행 창구]
      창구가 actor:
      · 창구는 한 번에 한 명씩만 응대 (actor isolation)
      · 손님 A가 "100만원 송금" 요청 → 창구원이 처리 중 → 송금 영수증을 안내판에 띄움
        (inFlight = task)
      · 손님 B가 같은 송금을 요청 → 안내판 보고 "어, A의 송금이 진행 중이네"
        → 영수증 도착하면 같은 결과 받음 (Task.value 공유)
      · 영수증 도착 후 안내판 치움 (defer { inFlight = nil })

      "창구가 한 번에 한 명씩 받는 것이 actor의 본질. 안내판은 도구."
      둘이 합쳐져야 "중복 송금 0건"이 달성됨.

      [가장 중요한 학습 포인트]
      "actor를 단순히 '락이 자동인 클래스'로 보는 단계에서 한 발 더 들어가서,
       actor는 메서드 진입 자체를 자연스럽게 직렬화해주는 동시성 단위라는 개념이 박혀야
       Swift 6 동시성이 풀린다."

  ## 심화 — Task 두 줄 깊이 읽기

  TokenRefreshCoordinator의 가장 헷갈리는 두 줄:

  ```swift
  let task = Task<String, Error> { try await block() }
  inFlight = task
  ```

  ### 첫 줄: `let task = Task<String, Error> { try await block() }`

  한 줄에 세 가지가 동시에 들어 있음:

  **(1) `Task<String, Error>` — 타입**
  Task = Swift Concurrency의 기본 단위. "한 덩어리의 비동기 작업"을 표현하는 객체.
  · `Success = String` → 성공 시 String 반환
  · `Failure = Error` → 실패 시 Error throw 가능
  → 의미: "문자열을 비동기로 만들어내거나 에러를 던지는 작업"이라는 타입

  **(2) `{ try await block() }` — 작업 내용 (클로저)**
  중괄호 안이 실제 할 일. 여기선 외부에서 받은 block 클로저를 await해서 실행하는 것이 일.

  **(3) `let task = Task { ... }` — ★ 즉시 실행 시작 ★ (가장 중요)**

  Task { ... } 를 변수에 대입하는 순간 클로저가 이미 실행되기 시작함.
  변수 task는 그저 "지금 막 시작한 작업의 손잡이(핸들)"를 보관할 뿐.

  일반 클로저와 비교하면 차이가 명확:

  ```swift
  // 일반 클로저
  let work = { try await block() }   // 저장만 됨. 안 실행됨
  let value = try await work()        // 여기서 처음 실행

  // Task
  let task = Task { try await block() }   // ⚠ 이미 실행 중!
  let value = try await task.value         // 결과 기다리기만 함 (실행은 이미 됐음)
  ```

  → "Task는 **레시피가 아니라 이미 시작된 요리**". let task = ... 자체가 "block() 실행 시작!"의 의미.

  ### 둘째 줄: `inFlight = task` — 결과가 아니라 "손잡이"를 게시

  자주 빠지는 오해:
  > "함수 내에서 실행되어 나온 결과를 공지하는 것"

  정정:
  > "**진행 중인 작업의 손잡이(=Task)**를 공지하는 것"

  이 시점엔 결과가 아직 없음. Task가 막 시작했을 뿐이라 네트워크 응답은 아직 안 옴.

  ### 시간순 그림

  ```
  t1  let task = Task { ... }    ← 작업 시작 (네트워크 요청 발사). 결과는 아직 없음
  t2  inFlight = task             ← "이 작업이 진행 중"이라는 손잡이를 게시. 결과는 여전히 없음
  t3  await task.value            ← 결과 도착할 때까지 기다리기 시작
  ... (네트워크 왕복 수백 ms) ...
  t9  결과 도착                   ← 이때 비로소 모든 awaiter가 결과 받음
  ```

  t2 시점엔 결과가 아직 없음. 그런데도 게시는 함. 이게 핵심.

  ### 왜 결과 없이 게시하나

  두 번째 호출자가 t6쯤(첫 호출자가 await로 suspend된 후) 들어왔을 때:

  ```swift
  if let existing = inFlight {            // ← 진행 중인 작업의 손잡이 발견
      return try await existing.value     // ← 같이 결과 도착을 기다림
  }
  ```

  게시된 건 "이 작업이 진행 중"이라는 사실 + 그 결과를 받아갈 손잡이.
  두 번째 호출자는 그 손잡이로 함께 기다림. 트럭이 도착하면(t9) 둘 다 동시에 결과 받음.

  ### 두 줄을 합치면

  ```swift
  let task = Task<String, Error> { try await block() }    // ① block을 비동기 작업으로 즉시 시작
  inFlight = task                                          // ② 손잡이를 actor 게시판에 꽂기
  ```

  한 문장 번역:
  "block을 비동기로 즉시 실행하기 시작하고, 그 작업의 손잡이를 게시판에 꽂아
   다른 호출자가 발견할 수 있게 만든다."

  순서: ① 시작 → ② 게시. 게시되기 전에 다른 호출자가 들어와 봐도 inFlight == nil이라
  자기도 Task를 만들 텐데, actor isolation 때문에 그런 일이 일어날 수 없음 (Q4 보강 참조).
  첫 호출자가 ②까지 마치고 다음 줄(await task.value)에서 suspend되어야 두 번째 호출자가
  actor에 진입.

  ### 송장 게시판 비유로 굳히기

  | 코드 | 비유 |
  |------|------|
  | Task { try await block() } | 택배 주문 = 트럭이 즉시 출발 (주문하는 순간 작업 시작) |
  | task 변수                  | 송장 번호 — 손에 든 추적 번호 |
  | inFlight = task            | 송장 번호를 회사 게시판에 적어두기 |
  | task.value 의 await        | 그 송장이 도착할 때까지 기다리기 |
  | 두 번째 호출자 발견          | 게시판 보고 "아, 이미 배송 중이네" → 같은 배송 결과 공유 |

  게시한 건 송장 번호(손잡이)지 택배 자체(결과)가 아님.
  손잡이를 들고 있으면 결과 도착 시 함께 받을 수 있다는 게 핵심.

  ### 자주 빠지는 함정 4가지

  1. **Task { ... }는 lazy하지 않음** — 만들면 즉시 시작. 클로저 변수와 정반대.
  2. **await task.value는 작업을 '시작'하지 않음** — 결과를 '기다리기만'. 작업은 Task 생성 시점에 이미 시작됨.
  3. **task 변수를 어디에도 안 저장해도 작업은 계속 실행됨** — Swift 런타임이 살려둠. 다만 결과를 받을 곳이 없으면 fire-and-forget.
  4. **Task.detached { ... } 는 부모 context를 끊고 시작하는 변형** — AppDelegate의 Task.detached가 그 케이스. 일반 Task { ... } 는 호출 위치의 actor/priority를 상속.

  ### 한 줄 격언

  > "Task = 손잡이고 결과는 별도. inFlight = task 는 **손잡이를 게시판에 꽂는 행위**.
  >  결과는 t9에 도착하면 손잡이를 든 모든 호출자가 함께 받음."

  이 분리가 박히면 다음 코드(await task.value, existing.value 등)가 자연스럽게 읽힘.
  Task 생성·게시·공유의 3단 구조가 보이기 시작.

  ─────────────────────────────────────────────────────────────────────
  ## 4개 자문 종합 평가

  | Q | 평가 | 핵심 |
  |---|------|------|
  | 1. Task<String,Error>?의 race 방지       | ⭐ 핵심 잡음 | "내부 캐시" 키워드 정확 |
  | 2. Task.value 여러 번 await             | ⭕ 결론 맞음 | "취소 전파" 가설은 잘못. await 가능성은 Task 기본 보장 |
  | 3. defer로 핸들 비우는 이유              | △ 절반 맞음 | "다음 호출자가 죽은 결과를 신선한 것으로 착각"이 정확 |
  | 4. 클로저 직렬화 메커니즘                 | ❌ 핵심 놓침 | actor isolation + inFlight 검사 + Task 캐시 세트의 합작 |

  가장 중요한 학습 포인트는 Q4 — actor isolation이 직렬화의 근본이라는 통찰.
  이게 박히면 다음 모든 actor 코드가 더 쉽게 읽힘.

  보너스 — 코드 밖 통찰

  @Sendable @escaping () async throws -> String 이 클로저 타입 시그니처 의미도 곱씹어보세요:
  - @Sendable — 이 클로저는 다른 actor/스레드로 건너갈 수 있다. 캡처하는 값들도 다 Sendable이어야 함
  - @escaping — 함수 반환 이후에도 살아남는 클로저
  - 합치면 "actor 경계를 안전하게 넘는, 나중에 실행될 비동기 작업"
